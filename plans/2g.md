# Bead 2g — Snapshot policy integration: explicit + on-close

## Scope

Wire `SnapshotService` into:

1. `SessionService.close` — take a snapshot inside the same effect that
   flips status to closed. Uses the existing `show`-style reduce path:
   read events, fold via `reduce()`, write snapshot. The new
   `sessions.last_snapshot_event_id` pointer is updated automatically
   by `SnapshotService.write`.
2. `SessionService.takeSnapshot` — explicit trigger for `cognit
   snapshot`. Idempotent: returns the existing latest row if a
   snapshot already exists for the latest event id.

## Files

- `packages/db/src/session-service.ts` (edit)
  - Import `SnapshotService` from `./snapshot-service`.
  - Add `SnapshotService` to the `Layer.Layer` R channel of
    `SessionServiceLive`.
  - Yield `SnapshotService` inside the layer effect.
  - In `close`: after appending `session_closed`, fold events, write
    snapshot. On snapshot failure, log warning but still succeed the
    close (the close already committed; orphan snapshots are not
    catastrophic).
  - New method `takeSnapshot(sessionId, actor?)`:
    - Reads the latest snapshot; if `event_count` matches current
      event count, return the existing row (no-op).
    - Otherwise read all events, reduce, write a fresh snapshot.

- `packages/db/src/layers/live.ts` (edit)
  - `SessionServiceLive` R-channel now includes `SnapshotService`; the
    `DbLive` factory must satisfy it. We provide `snapshotServiceLayer`
    alongside `sessionServiceLayer` so `DbLive` already includes both.

- `packages/db/test/session-service.test.ts` (edit) — add tests:
  - `close` writes a snapshot row.
  - `close` is idempotent on already-closed sessions.
  - `takeSnapshot` returns existing when up to date.
  - `takeSnapshot` writes a new snapshot when called after additional
    events.
  - `takeSnapshot` on unknown session returns `UnknownSession`.

## Out of scope

- Auto-N-event trigger (deferred per plans/phase-2.md; manual + on-close
  are ship-blocking).
- Tx-based atomicity between status update and snapshot write: the
  session status flip and the event append already share the close
  effect; the snapshot write is a separate insert. A crash between
  status update and snapshot write leaves a session whose
  `last_snapshot_event_id` is stale (points to an earlier event). The
  reducer handles this gracefully on `show` — it discards the stale
  snapshot and re-folds from scratch.
