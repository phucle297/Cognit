# Bead 1i — CLI: cognit append + cognit inbox

## Goal

Wire the last phase-1 piece so the user can write events by hand and
drain the inbox. After this bead:

```
cognit append --type observation_recorded --payload '{"text":"x"}' --session <id> [--actor name:type]
cognit inbox --process     # drain the current queue once
cognit inbox --watch       # start a chokidar daemon
```

## Command details

### `cognit append`

Flags:

- `--type <string>` — event type (must be in EVENT_TYPES).
- `--payload <json>` — either a JSON string or a path to a `.json` file
  (read its contents). The default is `{}`.
- `--session <id>` — required; the session id (ULID).
- `--actor <name:type>` — optional; default `cognit-cli:system`.

Validation: validate `--payload` parses as JSON. The schema check
happens in `EventStore.append`; surface a clean error if it fails.

Output: print the new event id, its type, and the session id.

### `cognit inbox`

Flags:

- `--watch` — start a long-running chokidar watcher. Use the existing
  `makeInboxWatcher` from `@cognit/db/inbox`. Resolves only when the
  watcher errors; tests must `child.kill()` after a short delay.
- `--process` — drain the current inbox queue once. Use
  `processInbox` (or equivalent — see what's exported from
  `@cognit/db/inbox`). Print the count of files processed and
  the count of errors.

For `--watch` and `--process`, the existing `inbox.ts` in db
exports what we need. If `processInbox` doesn't exist, use
`makeInboxWatcher` with a no-op chokidar wrapper OR add a small
`processInboxDir` helper to db/src/inbox.ts (a one-time drain that
reads every file in inboxDir, processes it, and returns counts).
