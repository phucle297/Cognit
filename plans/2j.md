# Bead 2j — End-to-end: snapshot+tail replay + resume-as-fork (the done_when)

## Goal

The phase-2 done_when test. It is the single E2E that proves the
whole stack works: append events → snapshot at boundary → resume
snapshot+tail → state matches expectation → close writes second
snapshot → CLI resume-as-fork creates a new session linked to the
parent.

After this bead, `cognit session resume <id>` rebuilds full session
state from `(last snapshot) + (events after it)`.

## Files

- `packages/db/test/reducer-integration.test.ts` (NEW) — the integration
  test that runs the full append → snapshot → resume sequence against
  a real (temp) SQLite DB. Asserts timeline length, hypothesis state,
  and snapshot row existence.
- `packages/cli/test/resume-e2e.test.ts` (NEW) — CLI E2E. Spawn the
  CLI, run `init`, `session create`, append a few events, close,
  then `session resume <id> --fork=true`. Assert the forked session
  has `parent_session_id` set and a `session_created` event was
  appended with that parent id.

DO NOT TOUCH any service files. This bead is test-only.

## The E2E test outline

```ts
// packages/db/test/reducer-integration.test.ts
describe("reducer integration — the done_when", () => {
  it("appends 100+ events, snapshots, replays tail, closes, resume-as-forks", async () => {
    // 1. setupProject + create session
    // 2. append(session_created), then 3x observation_recorded
    // 3. append 2x finding_created
    // 4. append hypothesis_created, hypothesis_weakened, hypothesis_rejected
    // 5. append experiment_created, experiment_completed
    // 6. append conclusion_proposed, conclusion_verified
    // 7. append decision_proposed, decision_accepted
    // 8. append 90 more observation_recorded  (total ~ 100+ events)
    // 9. call SnapshotService.takeIfDue(sessionId, 100, 100) → assert snapshot exists
    //     with event_count = 100
    // 10. append 5 more events
    // 11. SessionService.show(sessionId):
    //     - assert state.timeline.length === 105
    //     - assert state.snapshot_event_id is set
    //     - assert tail_event_count === 5
    //     - assert the hypothesis is in "rejected" state
    // 12. SessionService.close(sessionId) → assert 2nd snapshot exists
    // 13. SessionService.resume({idOrGoal: originalId, fork: true, ...}) →
    //     - assert forked session has parent_session_id === originalId
    //     - assert session_created event was appended with the parent
});
```

## Notes

- The reducer integration test is the hard one. Use the same test
  layer pattern as `session-service.test.ts` (DbConnection +
  EventStore + SessionService + SnapshotService + Logger). The
  `takeIfDue` needs a `build` function — pass `reduce` from
  `@cognit/core/reducer`.
- For the `takeIfDue` boundary test, total event count must be
  exactly 100 at the snapshot trigger. 14 hand-built events + 86
  loop iterations to hit 100; then 5 more.
- The CLI test reads the DB via `better-sqlite3` (already a transitive
  dep via @cognit/db). Importing it directly in a CLI test is fine;
  it's already in the lockfile.
- The 5 events appended after the snapshot are the "tail" — the
  reducer should only fold these on `show`.
