## Goal

Wire the `cognit` CLI to expose session lifecycle and the snapshot
trigger. After this bead, a user can:

```
cognit session create "find the leak"
cognit session list
cognit session show <id-or-goal>
cognit session pause <id-or-goal>
cognit session resume <id-or-goal> [--fork=true]
cognit session close <id-or-goal>
cognit snapshot [--session <id>]
```

All commands operate on the current project (findProjectRoot). They
print human-readable output (no JSON by default in phase 2).

## Command surface details

### `cognit session create "goal" [--parent id] [--actor name:type]`

Resolves `--actor` (default `cognit-cli:system`), calls
`SessionService.create({ projectId, goal, parentSessionId, actor })`,
prints the new session id and goal.

### `cognit session list [--status active|paused|closed]`

Resolves `--actor` and `--status`, calls
`SessionService.list({ projectId, status })`, prints a table:
`id | status | goal | created_at | events | last_snapshot`.

### `cognit session show <id-or-goal>`

Resolves the id/goal, calls `SessionService.show(sessionId)`, prints:

- Header: id, status, goal, parent_session_id, created_at,
  event_count, snapshot_event_id, tail_event_count.
- Sections (one per `SessionState` section that is non-empty):
  - Rejected hypotheses (count + ids + titles).
  - Verified conclusions (count + ids + text).
  - Accepted decisions (count + ids + text).
  - Observations (last 5).
  - Findings (last 5).
  - Timeline (last 10 events: type + first 80 chars of payload).

### `cognit session resume <id-or-goal> [--fork=true]`

Resolves `--actor`, calls `SessionService.resume`. Prints the new
session id (or the reopened id) and indicates fork vs reopen.

### `cognit session pause <id-or-goal>`

Resolves `--actor`, calls `SessionService.pause`. Prints the new
status.

### `cognit session close <id-or-goal>`

Resolves `--actor`, calls `SessionService.close`. Prints the new
status and the snapshot id (because close triggers a snapshot).

### `cognit snapshot [--session <id>] [--actor name:type]`

If `--session` is provided, uses it; otherwise uses the most-recent
active session. Calls `SessionService.takeSnapshot`. Prints whether a
new snapshot was taken or an existing one was returned, and the
event_count.

## Resolve helpers

`resolveSessionRef(ref: string, projectId: string, service: SessionService)`:

- If the ref matches `/^01[A-Z0-9]{22,}$/i` â†’ treat as id; call
  `getByGoalOrId({ id })`.
- Else â†’ treat as goal; call `getByGoalOrId({ goal, preferMostRecent: true })`.
- If `getByGoalOrId` returns `ambiguous: true`, print a warning to
  stderr (the service picked the most recent; user can retry with
  the id).

`resolveProjectId(layer)` â€” see the ProjectService note below. The
CLI does not know the project id up front; it calls
`ProjectService.ensure({ name })` which is idempotent: returns the
existing project row if a row with that name exists, otherwise
creates a new one with a fresh ULID.

## Project bootstrap (gap fill)

The current codebase has no production code that creates rows in
the `projects` table. Phase-1 tests insert directly. For the CLI to
work end-to-end, we need a `ProjectService` that the session CLI
calls on startup.

Add `packages/db/src/project-service.ts` (NEW):

- `ProjectService.ensure({ name: string }): Effect<ProjectRow, DbError>`
  â€” idempotent. If a row with `name = ?` exists, return it; otherwise
  generate a ULID and insert.
- `ProjectService.get(id: string): Effect<ProjectRow | null, DbError>`.
- `ProjectService.byName(name: string): Effect<ProjectRow | null, DbError>`.
- Export `ProjectService` + `ProjectServiceLive` from `db/src/index.ts`.
- Wire it into `DbLive` so the full live layer provides it.

The CLI's `cognit session create` and friends call
`ProjectService.ensure({ name: <project name from config> })` before
invoking `SessionService.create`. This way the project row always
exists when sessions are created. The project name comes from
`readConfig(configPath).project.name`.
